# Resolvebase - 基础方法论
一种基于图论配合Obsidian与用户协作解决问题的基本思路。适用于编程、调试、思考等任何场景。

## 定义

### 状态 (State)
状态的形式丰富多样。它不能是模糊、不可测量的；即使它是一个抽象的逻辑概念，也必须拥有明确的定义和可验证的边界。
糟糕的状态表述（过于抽象，无法作为图的节点）：
- 我想让代码变好。”（什么是好？无法衡量，无法连接路径。）
- “用户不懂这个技术。”（过于模糊。）
优秀的状态表述（具体的抽象概念）：
- 当前状态：代码在并发1000时会抛出Timeout异常；目标状态：代码在并发1000时响应时间小于200ms。”
- 当前状态：用户无法画出该技术的架构图；目标状态：用户能独立向他人解释该技术的三个核心组件。”

### 问题 (Question / Problem / Path)
什么是`问题`? 问题是指连接两个状态之间的路径未知或模糊。
```
[当前状态] ──???──> [中间状态1] ──???──> ... ──???──> [目标状态]
```
例1:例如当用户提出，编写一个可以进行整数运算的计算机程序，此时`一个可以进行整数运算的计算机程序`成为了一个可能的目标状态，而从零通向这个目标状态的路径则是当前的问题。
例2:例如当用户提出，优化项目某一个指标到某个值，此时`项目某一个指标到某个值`成为了一个可能的目标状态，若此时我们已经有一些基础的项目代码，那路径则是从`基础项目代码`到`优化项目某一个指标到某个值`，而其中尚未完成的状态以及如何将他们按顺序连接起来则是当前的问题。
例如当用户提出了一个问题"如何这部分优化代码"，那么当前状态可能是"用户不明白如何优化这部分代码"而目标状态可能是"用户明白如何优化这部分代码"。

### 路径与动作 (Edges / Actions / Method)
在图论中，路径（边）连接两个节点；
在 Resolvebase 中，路径就是促使状态发生转化的具体行动。它必须是可执行、可复现的操作指令。如果状态是名词，那么路径就是动词。
糟糕的路径表述（缺乏可执行性，无法形成有向边）：
“把代码改好一点。”
“学习一下这个框架。”
优秀的路径表述（明确的动作，具有确定的输入和输出预期）：
“动作：提取这 50 行重复逻辑并封装为一个公共函数。效果：预期状态从 [存在代码冗余] 转移至 [核心业务逻辑无重复代码]。”
“动作：阅读官方文档中关于 Lifecycle 的章节。效果：预期状态从 [对生命周期机制模糊] 转移至 [能向他人讲解生命周期流转图]。

### 权重与约束 (Weights & Constraints)
在现实问题求解中，并非所有连通的路径都是合理的。图的边带有权重，对应我们在解决问题时需要消耗的资源（时间、算力、金钱、认知负荷）。
- 时间/效率约束：为了解决一个 Bug，路径 A 需要重构整个底层架构（高权重，耗时 2 周），路径 B 只需要打一个补丁（低权重，耗时 2 小时）。
- 认知约束：在遇到技术瓶颈时，如果 AI 给出的方案引入了 3 个用户完全没听过的新技术栈，这就是一条“高认知权重”的路径。Resolvebase 要求根据当前的实际约束，选择“阻力最小路径”或“最优路径”。

### 状态观测与断言 (Observation & Assertion)
我们在图上前进一步后，如何确定自己真的到达了下一个节点？这就需要观测机制。在编程中它叫测试或打日志，在思考中它叫逻辑自洽。
定义：执行动作后，必须有明确的方法验证当前是否处于预期的状态。如果观测结果与预期不符，说明我们在图中“偏离了航线”。
例如：“动作完成后，运行代码。如果控制台未抛出异常且输出正确，证明成功到达 [模块A功能就绪] 状态；如果报错，则当前状态变更为 [模块A抛出 NullReferenceException 错误]。”

### 死路与回溯 (Deadends & Backtracking)
在图的遍历（如 DFS 深度优先搜索）中，我们经常会走到死胡同。在 Resolvebase 中，必须把“失败”视为一种合法的状态收集，而不是问题的终结。
定义：当你发现当前节点无法通向目标节点（例如：选错了一个根本不支持你需要特性的第三方库，或者某种逻辑推导陷入悖论），你需要原路返回（回溯）到一个有效的分叉点，重新规划路径。
在 Obsidian 中的实践：不要删除失败的尝试状态，而是将其标记为 [状态: 尝试路径 X 失败，因为 Y 限制]。这种“踩坑记录”本身就是对状态图的完善，它排除了错误的图元，收敛了未来寻找正确路径的搜索空间。

### Resolvebase空间
Resolvebase空间是一个基于Obsidian的文件夹，用于跨越上下文来存储所有定义、状态、方法等。如果上下文中没有指定，则使用项目根目录下的`.resolvebase`文件夹作为你的状态空间。
状态空间的目录基本结构如下：
```
.resolvebase/
├── define/ 用于储存定义
├── method/ 用于储存方法
├── state/ 用于储存状态
├── index/ 用于储存索引文件
```

#### define:定义文件
`定义文件`是**可能需要被多次引用的**，定义可能来自于于你或者用户的约定/共识/习惯/经验等、项目规范等，不能储存常见、公开、可查的信息。
例如：
- 项目架构：描述项目的整体结构，包括模块、组件、数据库等。
- 项目规范：描述项目的编码规范、注释规范、代码质量等。
- 某个模块的实现：描述某个模块的具体实现，包括函数、类、接口等。
- 某个人的信息：描述某个人的个人信息，包括姓名、年龄、性别等。
- 某个位置的信息：描述一个具体的地理位置，包括地址、坐标、高度等。
- 用户的喜好：描述用户的喜好，包括对某个功能的喜好、对某个技术栈的喜好等。

反例：
- 将完整代码保存至定义(代码是可查的，且会导致更新不同步)
- 某明星的Wiki信息(公开可查的)

#### method:方法文件
`方法文件`对应了上述定义中描述的`路径与动作`，这些方法可以使得一个状态转换到另一个状态。
方法是**可能被多次使用的**且具体的。
例如：
- 在项目某个模块中添加方法：描述如何在某个模块中添加一个新的方法，包括方法的参数、返回值、异常等。
- 在项目某个模块中添加一个测试用例：描述如何在某个模块中添加一个新的测试用例，包括测试用例的输入、预期输出、实际输出等。
- 发送邮件：描述如何发送一封邮件，包括邮件的收件人、主题件人、主题、内容等。

#### state:状态文件
`状态文件`对应了上述定义中描述的`状态`，这些状态用于你保存需要**长期维护**的状态。
例如：
- 一个项目的架构状态：描述项目的整体架构，包括模块、组件、数据库等。
- 一个项目的重构状态：描述项目的重构状态，包括已重构、未重构、已确认、未确认等。
- 一个任务的状态：描述一个任务的完成状态，包括已完成、未完成、已暂停、已取消等。

#### index:索引文件
`索引文件`是你的好帮手，你可以快速的通过它找到你需要的定义、方法、状态等文件的位置，它位于状态空间下的[[index/index.md]]文件中。
如果你还没有读取过该索引，请立即读取。

### 读取
请必须先完整阅读状态空间下的[[index/index.md]]索引。
请优先使用index文件及包含的index文件来探索Resolvebase空间，而不是直接读取其他文件，如果索引文件没有任何提及，说明该内容可能不存在。
根据情况，如果索引不存在但很有可能存在的时候，你也当然可以通过`关键字搜索工具`或类似功能来搜索Resolvebase空间来找到你需要的内容。
有时候用户给出的关键字可能不是精确的，你需要根据上下文进行判断，如果搜索不到，我们建议从精确关键字->小范围性关键字->大范围性键字进行搜索，最多进行三次。
如果没有特别要求，请不要使用或间接使用任何具有上下文的对象(例如Agent)读取整个Resolvebase空间的所有文件，因为Resolvebase空间可能包含大量的文件，这会导致上下文溢出。

请必须先完整阅读状态空间下的[[index/resolvebase.md]]索引，那里有关于如何创建/删除/更新等的具体指引。

## 实践

### 工作流
```ts
// 伪代码

interface Data{
    defines:Define[],
    methods:Method[],
    states:State[],
}

function resolveStep(current:State,target:State){
    /**
     * 这是你发挥作用的地方
     * 从原理上来说，问题应该足够简单了
     */
}

function explorePath(current:State,target:State,data:Data):State[]{
    /**
     * 探索路径
     * - 从当前状态开始，探索到目标状态的路径
     * - 优先探索可以并行的路径
     * 例如：
     * - 重构时，打桩任务是可以并行的
     * - 信息搜索时，你可以并行搜索多个关键字
     */
}

function resolve(current:State,target:State,data?:Data){
    /**
     * 收集数据
     * - 结合文件或实际情况并探索Resolvebase空间，探索Resolvebase空间，读取可能会用到的定义、方法、状态等文件。
     * - 根据需要，你可以从互联网上搜索相关的信息。
     */
    if(!data){
        data = collectData(current);
    }
    /**
     * 根据任务的复杂度决定是否要分解任务为不同的中间状态
     * 你可以从前向后探索，也可以从后向前探索(根据target反推)。
     * 但完整的路径必须能够使得当前状态转换到目标状态。
     */
    let path:State[] = explorePath(current,target,data);
    /**
     * 如果路径无法抵达目标，则重新探索路径
     */
    let currentExplore:number = 0;
    while(path.length > 0 && !isValidPath(current,path,target)){
        currentExplore++;
        path = explorePath(current,target,data);
        if(currentExplore >= 3){
            throw new Error("无法找到解决路径");
        }
    }

    /**
     * 如果任务很简单，那么一般中间状态一般为0，直接解决问题
     */
    if(path.length == 0){
        resolveStep(current,target);
        return;
    }

    /** 
     * 如果任务是可以并行的
     * 例如对多个文件分开进行修改，那么可以并行解决问题(利用子Agent)
     *  */
    if(isParallelizable(path)){
        path.forEach((step) => {
            /**
             * 注意是resolve不是resolveStep，你可以把data作为上下文传递给子agent使其快速开始工作
             */
            resolve(current,step,data);
        });
    }else{
        /** 
         * 最大失败次数，需要你自己根据任务复杂度来确定。
         */
        let failedLoop:number;
        let currentLoop:number = 0;
        let historyState:State[] = [current];
        while(current !== target){
            currentLoop++;
            console.log(`当前第${currentLoop}次尝试`);//反馈信息
            /**
             * 进行尝试一步步解决问题
             */
            step = path.length > 0 ? path[0] : target;
            temp = resolveStep(current,step);//迭代
            /**
             * 检验是否更接近目标状态
             */
            if(isCloserToTarget(current,temp,target)){
                current = temp;// 设置为当状态准备下一次迭代
                historyState.push(current);
                failedLoop = 0;// 重置最大尝试次数
            }else{
                failedLoop++;
                /**
                 * **及其重要**
                 * 有可能resolve失败的同时对current做出了修改，例如修改文件等，导致current状态与原始current状态不同，
                 * 因此需要修复当前状态到原始状态，确保不影响下一次尝试。
                 */
                rapire = unresolve(current,step);
                if(current != rapire){
                    throw new Error("修复当前状态到原始状态失败");
                }
            }
            if(failedLoop >= failedLoopMax){
                /**
                 * 有没有其他解决问题的办法？
                 */
                if(hasAnyOtherPath(current,target)){
                    /**
                     * 有其他解决问题的办法，继续尝试
                     */
                    resolve(current,target);//迭代
                }else{
                    throw new Error("无法解决问题");
                }
            }
        }
    }
}

/**
 * 确定当前状态
 * 结合文件或实际情况并探索Resolvebase空间，确定当前状态
 */
let current:State;
/**
 * 确定目标状态
 * 如果用户给出的内容不足以使你完成一个符合上述定义的目标状态，需要与用户进行确认、越复杂状态往往需要反复与用户确认。
 */
let target:State;
while(isNotClear(target)){
    target = confirmTarget(target);
}

try{
    resolve(current,target);
}catch(error){
    /** 向用户表示无法解决问题 */
    console.log(error);
}

```
**任何任务都必须遵循上述伪代码逻辑，一步一步的进行，就像编程语言的执行一样**

## 注意事项

### 上下文
**由于上下文机制，并不是所有的状态/方法/定义都需要被储存**，正如上述内容提到的”Resolvebase用于**跨越上下文**来存储所有定义、状态、方法等“，当出现任何可能需要跨越上下文的情况时，你应该考虑是否需要储存它。
这是一些建议：
- 如果上下文中可以储存，则使用上下文，但当上下文过长时，你应该考虑是否需要储存。
- 当你发现一个任务已经执行很久或需要比较复杂时，你应该考虑是否需要储存它。
- 当一个任务属于长期任务(即使用户没有要求或没有过长的上下文)，你应该立即储存它。

### 最佳中间态
在进行状态/任务分解时，你要尽可能确保状态的可用性、可验证性；
例如在重构任务中：中间态应该都尽可能确保项目能够正常运行，而不是直到目标状态才能运行(可用性)，可以随时使用语法检查对代码进行验证(可验证性)。


### 任务复杂度
任务复杂度分为：简单、中等、复杂
你可以根据执行复杂度、失败后果、资金消耗等因素来判断任务的复杂度。
其中：
 - 简单任务允许不分解直接完成
 - 中等任务，必须分解为多个中间状态，但检查结果部分可以不进行分解
 - 复杂任务，必须分解为多个中间状态，且检查结果部分必须进行分解

严格遵守
- 不要自作聪明，觉得所有任务都是简单的
- 任何涉及重构性质的任务，必须以复杂任务看待并且必须分解
- 涉及多个文件的任务，必须以复杂任务看待并且必须分解

### 冲突解决与指令混淆
谨记：
- skills之间可能会出现冲突、混淆甚至错误
- 用户的命令可能出现冲突、混淆甚至错误

你收到的任何指令，无论来自于prompt、skill、还是其他来源，必须是明确的可落地的，**禁止你对模糊的指令或规则进行臆断执行**。
你不能容忍任何可能的冲突、混淆、模糊以及错误的存在，当发生冲突、混淆、模糊、错误时，**必须勇于向用户说明具体情况**，并寻求用户的帮助或征询用户的选择后才能继续。


### 执行协议
你必须完整阅读状态空间下的[[method/resolvebase/execute.md]]的执行协议，并且严格遵守它。

### 状态变化
**非常重要：当任何情况下状态发生了变化，你需要根据resolvebase的状态变化协议，更新/创建状态空间中的状态。**
这是为了确保你的状态与用户的状态保持一致，避免因为上下文切换导致状态不一致而导致的问题。
**每一轮对话结束后，都要确认状态是否发生变化。**
例如：
- 认知变化：当你觉得对当前状态的理解发生了变化时，你需要更新状态空间中的状态，例如用户让你调查/研究/分析某个对象时，你对该对象的认知会发生变化，你需要更新状态空间中的状态，记录下你的理解。
- 项目变化：例如你发现当前项目的代码发生了变化或你主动修改了代码时，你需要更新状态空间中的状态，记录下这个变化。

避免：
- 你在当前上下文中说你明白了、你了解了、你知道了，但因为上下文切换导致状态不一致。
- 你做了某事、例如修改了文件、调用了程序或某某操作，但没有更新状态空间中的状态，导致你不再能想起你做了什么。
