# 状态文件

## 本质

状态文件是 Resolvebase 空间中记录**运行时快照**的单元，对应代码中的运行时变量。它描述某个实体在当前时刻的运行状况。

## 核心特征

- **可验证边界**：状态必须是明确、可验证的。它不能是模糊的感受（"代码不好"），而应是可客观判断真伪的断言（"模块 X 的单元测试覆盖率为 0%"）。
- **有目标**：每个状态文件指向一个期望状态（目标），这是问题求解的方向。若目标尚不明确，应显式标注"待确认"。
- **有变更历史**：状态通过日志记录每次变更，日志是追加式的、不可修改的，每条记录带有日期。
- **跨上下文追踪**：状态文件存在的核心理由是，该状态需要跨越多次对话追踪，不会在单次对话内终结。若仅在本次对话内有效，使用上下文即可。

## 内容边界

状态文件的内容是**观测记录**——当前快照（可验证断言）和变更日志（按时间排列的事件序列）。它不是 [[define/resolvebase/define|定义]]（不可变知识单元），也不是 [[define/resolvebase/method|方法]]（操作流程）。

## 引用约束

状态文件可引用 [[define/resolvebase/define|定义文件]]、[[define/resolvebase/method|方法文件]]和其他 [[define/resolvebase/state|状态文件]]，**禁止引用**[[define/resolvebase/index|索引文件]]。

## 上下文章节

每个状态文件必须包含 `## 上下文` 章节。当当前状态涉及的细节尚未固化为 [[define/resolvebase/define|定义]]，但又必须在清空上下文后仍能还原理解时，在此缓存提炼后的关键内容；若所有细节已由 [[define/resolvebase/define|定义]]完整覆盖，则用 `[[link]]` 引用即可。该章节是状态文件从运行时上下文"落盘"的唯一出口。

内容约束：
- 不得照抄源文件原文，必须提炼为关键断言
- 有可复用的 [[define/resolvebase/define|定义]]时，优先用 `[[link]]` 引用，不能复用才内联
- 不得包含可被独立查找到的内容原文（源码片段→写文件路径+行号，文件名/目录→写路径，公开文档原文→写URL+标题，可网络搜索的知识→写搜索关键词）。只有当前上下文独有的分析产物（推理结论、综合判断、设计决策）才可内联
- 总长度不超过 1000 字符
